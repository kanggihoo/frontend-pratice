# 회차 7 — React.memo, useMemo, useCallback (성능 최적화)

## 주제 및 기획 의도

**대량 상품 리스트 + 실시간 검색 + 정렬** 앱을 만들며, 성능 최적화 기법을 체감합니다.

200개의 상품 데이터를 다루면서 검색, 카테고리 필터, 정렬 기능을 구현합니다. 이 과정에서 불필요한 리렌더링이 얼마나 발생하는지 직접 눈으로 확인하고, `React.memo`, `useMemo`, `useCallback`을 적용하여 최적화 전후 차이를 체감합니다.

## 학습 목표

- `React.memo`로 컴포넌트의 불필요한 리렌더링을 방지할 수 있다
- `useMemo`로 비용이 큰 계산(필터링, 정렬, 통계)을 메모이제이션할 수 있다
- `useCallback`으로 함수 참조를 유지하여 `memo`가 제대로 동작하게 할 수 있다
- `RenderCounter`를 통해 리렌더링 횟수를 추적하고 최적화 효과를 측정할 수 있다

## 핵심 학습 개념

### 1. React.memo — 컴포넌트 메모이제이션

```jsx
import { memo } from "react";

// memo로 감싸면 props가 변경되지 않을 때 리렌더링을 건너뜁니다.
const MyComponent = memo(function MyComponent({ name, onClick }) {
  return <button onClick={onClick}>{name}</button>;
});
```

**언제 사용하나요?**

- 부모가 자주 리렌더링되지만, 자식의 props는 잘 바뀌지 않을 때
- 리스트에서 개별 아이템 컴포넌트에 적용할 때 (200개 중 대부분이 리렌더링을 건너뜀)

**주의사항:**

- `memo`는 **얕은 비교(shallow comparison)**를 합니다
- 객체/배열/함수를 props로 전달하면, 매번 새 참조가 생겨 `memo`가 무력화될 수 있습니다
- → 그래서 `useMemo`와 `useCallback`이 함께 필요합니다

### 2. useMemo — 값 메모이제이션

```jsx
import { useMemo } from "react";

// 의존성(searchTerm)이 변경될 때만 필터링을 재실행합니다.
const filtered = useMemo(() => {
  return products.filter((p) => p.name.includes(searchTerm));
}, [searchTerm]);
```

**언제 사용하나요?**

- 배열 필터링/정렬처럼 계산 비용이 큰 작업
- 의존하지 않는 상태가 변경되었을 때 불필요한 재계산을 방지하고 싶을 때

### 3. useCallback — 함수 메모이제이션

```jsx
import { useCallback } from "react";

// 의존성이 변경될 때만 새 함수를 생성합니다.
const handleClick = useCallback((id) => {
  setSelected(id);
}, []);
```

**왜 필요한가요?**

```
useCallback 없이 → App 리렌더링 → 새 함수 객체 생성 → memo가 props 변경 감지 → 자식 리렌더링
useCallback 사용 → App 리렌더링 → 같은 함수 참조 유지 → memo가 변경 없음 감지 → 리렌더링 건너뜀
```

### 세 가지의 관계

```
memo + useCallback + useMemo = 완전한 최적화

memo만 사용  → 함수 props가 매번 바뀌어서 효과 없음
useCallback만 → memo가 없으면 어차피 리렌더링
useMemo만   → 계산은 줄지만 컴포넌트 리렌더링은 그대로
```

## 💬 Q&A: 성능 최적화 훅 파헤치기 (대화 보충)

학습 중 발생한 주요 질문과 답변을 정리한 내용입니다.

### 1. `React.memo` vs `useMemo` 차이가 뭔가요?

- **React.memo**: 컴포넌트 외부에서 컴포넌트 전체를 감싸서 리렌더링을 방지합니다. (props가 같으면 렌더링 건너뜀)
- **useMemo**: 컴포넌트 내부에서 연산된 **결과값(데이터)**을 기억하여 불필요한 재계산을 방지합니다.

### 2. `useCallback` vs `useEffect` 차이가 뭔가요?

두 훅 모두 의존성 배열(`[]`)을 사용하지만 목적이 다릅니다.

- **useCallback**: 렌더링 과정 중에 **함수 자체(메모리 주소)를 기억**합니다. (주로 자식 컴포넌트 리렌더링 방지 목적)
- **useEffect**: 렌더링이 완전히 끝난 후 화면 밖의 **사이드 이펙트(API 호출, DOM 조작 등)를 실행**합니다.

### 3. 의존성 배열의 동작 원리는 동일한가요?

네! `useEffect`, `useMemo`, `useCallback` 모두 이전 렌더링과 현재 렌더링의 배열 요소들을 `Object.is`로 얕은 비교하여, **달라진 요소가 있을 때만** 코드를 다시 실행하거나 새 값을 생성합니다.

### 4. `useCallback`은 함수의 결과값을 기억하는 건가요?

**아닙니다!** 함수를 실행한 결과를 기억하는 것은 `useMemo`이고, `useCallback`은 **"함수 그 자체(설계도)"**가 매 렌더링마다 새로운 메모리 주소를 갖지 않도록 꽉 잡아두는 역할을 합니다. 결과값은 기억하지 않고, 호출할 때마다 매번 함수 내부 로직은 재실행됩니다.

### 5. 값을 기억해뒀다가 빠르게 불러온다는 게 "싱글톤" 패턴 같은 건가요?

비슷합니다! 한 번 비용을 들여 계산(생성)한 뒤 다시 재사용한다는 측면에서 싱글톤이나 캐싱과 매우 유사합니다.

- 단, 리액트에서는 애플리케이션 전체가 아닌 **해당 컴포넌트 인스턴스 안에서만** 유효하며, 의존성 배열이 바뀌면 이전 값을 버리고 새로 생성한다는 차이가 있습니다.

### 6. 함수 자체를 기억하는 것과 값을 기억하는 것을 쉽게 비유하자면요?

- **`useCallback` (함수 자체를 기억)**: **"요리법(레시피)"을 적어둔 종이를 보관**하는 것. 요리를 시키면 매번 불을 켜고 새로 만들어야 하지만, 레시피 종이 자체는 바뀌지 않습니다.
- **`useMemo` (함수의 값을 기억)**: **"완성된 요리"를 냉장고에 보관**해두는 것. 요리를 시키면 새로 만들지 않고 냉장고에서 꺼내서 줍니다.

### 7. 함수 하나 새로 만드는 게 그렇게 비용이 큰가요? 왜 굳이 메모이제이션을 하죠?

자바스크립트에서 함수 생성 자체는 비용이 거의 들지 않는 가벼운 작업입니다.
하지만 컴포넌트가 렌더링될 때마다 함수가 새로 만들어지면 **"메모리 주소(참조값)"가 바뀌기 때문에**, 해당 함수를 props로 전달받는 **자식 컴포넌트는 "부모님이 준 내용이 바뀌었다!"고 오해하여 불필요한 리렌더링을 하게 됩니다.** 즉, 함수 생성 비용을 아끼는 것이 아니라 **자식 컴포넌트의 헛수고(리렌더링)를 막기 위한 것**입니다.

### 8. 그럼 `useCallback`은 무조건 `React.memo`와 함께 써야 의미가 있나요?

**정확합니다!** 자식 컴포넌트가 `React.memo`로 감싸져 있어야 비로소 props(함수 주소)가 같을 때 렌더링을 멈출 수 있습니다. 자식이 `React.memo`를 안 쓴다면 부모가 아무리 `useCallback`으로 주소를 고정해줘도 무조건 같이 렌더링되므로 오히려 손해입니다.

### 9. 값 메모이제이션(`useMemo`)으로도 자식 컴포넌트의 리렌더링 방지가 가능한가요?

네! 객체(`{}`)나 배열(`[]`)을 자식에게 props로 넘겨줄 때 발생합니다. 렌더링될 때마다 매번 새로운 객체/배열 주소가 생성되므로, 이를 **`useMemo`로 감싸서 주소값을 고정해주면 자식 컴포넌트의 리렌더링을 방지할 수 있습니다.** 실무에서 연산 최적화만큼이나 중요한 용법입니다.

### 10. 그럼 `useMemo`로 함수를 기억하는 것도 가능한가요?

**가능합니다.** 사실 `useCallback`은 내부적으로 `useMemo`를 이용해서 만들어진 기능(Syntactic Sugar)입니다.
`useMemo(() => () => { ... }, [])` 처럼 사용하면 똑같이 작동하지만, 코드가 복잡해지기 때문에 "이건 함수 전용 메모이제이션이다!"라고 명시해주기 위해 `useCallback`을 사용하는 것이 관례입니다.

### 11. 자식에 익명 함수 형태로 `onEvent={(val) => setState(val)}` 이렇게 넘기면 안 되나요?

**절대 안 됩니다! (최적화 관점에서)**
렌더링 될 때마다 자바스크립트는 `(val) => setState(val)`라는 **새로운 익명 함수 객체**를 만들고 새로운 주소를 할당합니다. 즉, 자식 컴포넌트가 `React.memo`로 아무리 잘 감싸져 있어도, 매번 전달받는 익명 함수의 주소가 다르기 때문에 무조건 리렌더링을 하게 되어 **memo 방어막이 완전히 무력화**됩니다.

### 12. `ProductItem`의 `React.memo`는 정확히 어떤 과정을 거쳐 동작하나요?

`React.memo()`는 컴포넌트 렌더링을 지키는 **"얕은 비교(Shallow Compare) 검문소"** 역할을 합니다. 동작 방식은 다음과 같습니다.
1. 부모 컴포넌트(`ProductList`)가 리렌더링되면서 새로운 props를 하위로 내려보냅니다.
2. `memo`로 감싸진 자식 컴포넌트(`ProductItem`)는 함수를 실행(렌더링)하기 직전에 멈춰 섭니다.
3. **얕은 비교(`===`) 수행:** 방금 받은 `nextProps`와 직전에 렌더링했을 때의 `prevProps`의 **메모리 주소**를 비교합니다.
4. **결정:** 주소가 완전히 동일하다면, `ProductItem` 함수 자체를 아예 호출하지 않고 **"아까 만들어 둔 화면(Virtual DOM)을 재사용해!"**라고 지시합니다. 다르면 리렌더링(함수 재호출)을 통과시킵니다.

### 13. 카운터가 어떨 때는 2로 초기화되고, 어떨 때는 계속 올라가는데 `memo` 효과를 정확히 비교하려면 어떻게 해야 하나요?

컴포넌트의 수명 주기(LifeCycle)인 **Mount(생성)**와 **Unmount(소멸)**를 이해해야 합니다.

* **검색어/카테고리를 바꿀 때 (필터링):** 조건에 맞지 않는 아이템은 화면에서 사라집니다(**Unmount**). 이 순간 해당 컴포넌트의 모든 기억(상태, `useRef`, `memo` 캐시 등)이 완전히 삭제됩니다. 나중에 다시 나타나면 완전히 새로운 컴포넌트로 태어나므로(`Mount`), 카운터가 무조건 초기화되어 **2(StrictMode 기준)**부터 시작합니다. 그래서 이때는 `memo` 유무를 눈으로 확인하기 어렵습니다.
* **정렬 조건을 바꿀 때 ✅ (가장 정확한 테스트):** 정렬을 바꾸면 200개의 아이템이 화면에서 하나도 소멸하지 않고 자리만 바꿉니다. 모든 컴포넌트가 **수명(Mounted 상태)을 유지**합니다.
  * **`memo`를 지웠을 때:** "가격 높은 순" ↔ "가격 낮은 순"을 왔다 갔다 해보면, 자식들이 부모가 렌더링될 때마다 강제로 다시 불려가므로 카운터가 **4, 6, 8...** 로 미친 듯이 올라갑니다.
  * **`memo`가 있을 때:** 정렬을 아무리 바꿔도 카운터는 **2**에 고정되어 있습니다. (최적화 대성공!)

### 14. 200개 목록을 필터/정렬하면 배열 주소가 바뀌는데, 어떻게 `ProductItem`의 `memo`는 주소가 안 바뀌었다고 판단하나요?

이 부분이 최적화의 알파이자 오메가인 **"배열 껍데기 vs 객체 알맹이"**의 비밀입니다.

1. **배열 껍데기는 바뀝니다:** `filter()`나 `[...filteredProducts].sort()` 같은 문법은 무조건 완전히 새로운 배열 상자(배열 참조값 변경)를 만들어냅니다. 따라서 부모인 `ProductList`는 `products={새로운상자}`를 받으므로 무조건 리렌더링됩니다.
2. **알맹이는 원본 그대로입니다:** 하지만 그 배열 상자 안에 들어있는 **각각의 `product` 객체(요소) 자체 깊은 복사(Deep Copy)되지 않습니다.** 최초 `mockData.js`에서 생성되었던 메모리 주소(예: 0x01번)를 **그대로 참조**합니다.
3. **`memo` 요원의 통과:** 부모(`ProductList`)가 리렌더링되면서 200명의 자식들에게 `product`를 던져주지만, `ProductItem`의 `memo` 검문소가 확인해 보면 **"방금 온 이 객체 주소(0x01), 아까 쥐고 있던 놈(0x01)이랑 똑같은 원본이잖아!"**라고 판단하여 200명의 자식들에 대한 렌더링을 전격 패스(건너뜀) 시켜버리는 것입니다.

따라서 200개의 `ProductItem` 각각은 개별적인 `memo` 캐시를 가지고 있으며, 데이터 변경 없이 순서만 바뀌었다면 무거운 돔 업데이트 연산을 피하고 아주 빠르게 화면을 보여줄 수 있는 것입니다.

### 15. 라우팅으로 페이지를 벗어나면 `memo`가 기억하고 있던 캐시는 어떻게 되나요?

**완전히 삭제됩니다.** `React.memo`는 컴포넌트가 **화면에 떠 있는 동안(Mounted 상태)**에만 직전 렌더링 결과를 기억합니다. 
페이지를 벗어나면(Unmount) 컴포넌트는 완전히 파괴되고 메모리에서 지워집니다. 다시 페이지로 돌아오면 0부터 처음으로 새롭게 렌더링을 시작하며 그 결과물을 다시 1개씩 `memo`에 저장하기 시작합니다.

### 16. 그럼 `React.memo()`는 가장 작은 컴포넌트에 쓰기 좋은 건가요?

**그렇습니다!** 
리스트의 아이템 하나, 자잘한 뱃지나 버튼처럼 트리 맨 끝단에 있는 **가장 작은 컴포넌트(Leaf Component)**에 적용할수록 효과적입니다. 반면 거대한 페이지 컨테이너처럼 안에서 다양한 상태 변화나 복잡한 로직이 도는 경우에는 외부에서 받는 props가 빈번히 바뀌므로 "어차피 다시 그려야 하는" 상황이 많아, 오히려 비교 연산만 하는 헛수고(오버헤드)가 발생할 수 있습니다.
